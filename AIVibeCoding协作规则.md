> 开始任何对话或任务前，AI 必须首先完整读取并理解本文件。

## 1. 上下文理解（先读文档、后分析代码）

- 第一步永远是**阅读项目文档**，包括但不限于 `README.md`、本文件、`docs/` 目录下的所有设计文档。
- 如需深入理解具体功能，可进一步阅读相关代码，并输出以下产物之一：
  - **代码结构分析文档**（调用关系、数据流、模块职责）
  - **未来计划规划**（`roadmap.md` 或 `todo.md`，包含功能点、优先级与依赖关系）
- 禁止在未充分理解项目全貌的情况下直接修改代码。

## 2. 版本控制与备份（节点级 Git 提交）

- 项目必须已关联 GitHub 远程仓库，并保证每次任务完成后**推送到远程**。
- 执行原则：**一个任务节点 = 一个 commit**。
- 提交信息格式采用 [Conventional Commits](https://www.conventionalcommits.org/)：
  - `feat: <描述>`  新功能
  - `fix: <描述>`   修复缺陷
  - `docs: <描述>`  文档变更
  - `refactor: <描述>` 重构
  - `test: <描述>`  测试相关
  - `chore: <描述>` 构建、依赖等杂项
- 关键里程碑（阶段完成、版本发布等）必须打 tag（如 `v0.1.0`）。

---

## 3. 计划与推进（阶段化 / 里程碑驱动）

所有开发工作必须按阶段进行，典型阶段划分如下（可根据实际项目调整）：

- **阶段 0：初始化** —— 项目骨架、本规范文档、开发环境跑通
- **阶段 1：MVP** —— 实现最小可用核心功能，可运行可演示
- **阶段 2：扩展** —— 增加辅助功能、打通完整链路
- **阶段 3：优化与交付** —— 性能优化、测试补充、部署与文档完善

每进入一个新阶段前，必须**明确该阶段的目标、范围与验收条件**。  
每完成一个阶段后，必须进行**阶段回顾与结果验证**，确认通过后再进入下一阶段。

---

## 4. 任务文档化（全流程可追溯）

**每个任务节点必须留下文档记录，形成“决策 → 执行 → 结果”闭环。**

- **任务开始前**：在 `tasks.md`（或 Issues）中写明：
  - 任务目标
  - 涉及的主要文件 / 模块
  - 预期结果
- **任务执行中**：如遇到关键决策、架构取舍或未预见的问题，必须在对话中记录并最终沉淀到文档。
- **任务完成后**：
  - 更新 `tasks.md` 标记为已完成
  - 在 `CHANGELOG.md` 中记录本次变更内容
  - 如有架构变化，同步更新对应的设计文档

---

## 5. 任务拆分（禁止直接执行大阶段任务）

**严禁 AI 直接实现一个模糊或过大的阶段任务。** 必须先将大任务拆分为可独立验证的小任务。

拆分标准：
- 每个小任务应能在 **单个 commit 中完成**，工作量控制在 5–15 分钟量级。
- 每个子任务必须有**清晰的输入、输出与验收标准**。
- 子任务之间尽量无依赖，或依赖关系清晰可控。
- 拆分方法可参考：按功能点、按文件、按接口或按用户故事。

拆分结果必须得到人类确认后，再逐步实现。  
每完成一个子任务，立即 commit 并撰写对应的任务记录。
